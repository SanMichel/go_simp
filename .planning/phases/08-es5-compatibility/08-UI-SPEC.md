## Design System

| Property | Value |
|----------|-------|
| Tool | none — Go `html/template` + HTMX |
| Preset | not applicable (no shadcn) |
| Component library | none — hand-written Go templates + CSS |
| Icon library | none — emoji-only (📦 ✅ ⬅️ 🔍 📲 etc.) |
| Font | System stack: `-apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, Helvetica, Arial, sans-serif` |

### Context
This is a warehouse scanning application using Go server-rendered templates with HTMX partials. There is no React/Vite/Node ecosystem. No npm dependencies. No CSS framework. Design is minimal and utilitarian — targeting low-end warehouse devices with hardware barcode scanners.

## Interaction Contract

### File Architecture

The `templates/atividades/` directory will contain:

| File | Type | Role | Scripts Loaded |
|------|------|------|----------------|
| `atividades-login.html` | HTML template | Isolated login page for `/atividades/login` route | `atividades-utils.js`, `atividades-login.js` |
| `atividades.html` | HTML template | Main scanning SPA for `/atividades` route | `atividades-utils.js`, `atividades-scan.js`, `atividades-consulta.js`, `htmx.min.js` |
| `atividades-utils.js` | ES5 JS | Shared utilities (XHR wrapper, escHtml, showLoader, formatDate, playBeef) | — |
| `atividades-login.js` | ES5 JS | Login form handler (XHR to `/api/auth/login`) | — |
| `atividades-scan.js` | ES5 JS | Scanning workflow state machine, screens, scan loop, finalize | — |
| `atividades-consulta.js` | ES5 JS | Product consultation (by EAN code and by description) | — |

### Screen State Machine

Unchanged from existing `app.js`:

```
/atividades/login  →  screen-start  →  screen-scanning  →  screen-predio-switch
                           ↑                ↓                    ↓
                           |          screen-consulta       screen-report
                           |                ↓                    |
                           └────────────────┴────────────────────┘
```

Screen transitions use `showScreen(name)` — ES5 conversion of the existing function.

### Login Flow (new `/atividades/login` route)

1. User hits `/atividades` unauthenticated → Go handler checks `currentUser()` → redirects to `/atividades/login`
2. `atividades-login.html` renders with login form
3. `atividades-login.js` handles form submit:
   - XHR POST to `/api/auth/login` with username/password JSON
   - On success: `window.location.href = "/atividades"` (full page load to get fresh template with session)
   - On 401: show error message in `#login-error` element
   - On network error: generic error message

### Session Flow

1. `atividades-scan.js` on load: XHR GET `/api/auth/me` to validate session
2. If 401: redirect to `/atividades/login` (replaces old reauth modal behavior)
3. User data stored in `localStorage` for state persistence

### Scanning Flow (unchanged behavior)

1. Start screen: select company/location, enter street/building, submit → XHR POST to `/api/atividades/iniciar`
2. Scanning screen: scan EAN code → XHR GET `/api/produtos/ean/{code}` → display/validate
3. Building switch: if scanned address differs → predio-switch screen
4. Consulta: toggle between EAN code scan and description search → XHR GET `/api/produtos/consulta` or `/api/produtos/consulta?q=`
5. Finalize: XHR POST to `/api/atividades/finalizar` with scanned data → report screen

### ES5 Interaction Contract

All JS interaction patterns in the 4 new files MUST follow:

| Constraint | Rule |
|-----------|------|
| Variable declarations | `var` only — no `const`, no `let` |
| Functions | `function name() {}` or `function() {}` — no arrow functions |
| Async patterns | XMLHttpRequest with callbacks — no `async`/`await`, no `fetch`, no Promises |
| String building | String concatenation only — no template literals |
| Optional chaining | `&&` guard pattern — no `?.` |
| Array iteration | Index-based `for (var i = 0; i < arr.length; i++)` loops — no `for...of`, `forEach`, `.find()`, `.some()`, `.includes()`, `.filter()` — use `indexOf` for existence checks |
| Spread/rest | `arr.slice(0)` for copies — no `[...arr]`, no `{...obj}` |
| Default params | Manual `undefined` check — no `function(x = 5)` |
| Classes | Not used — plain objects and functions only |
| DOM methods | `document.getElementById()`, `element.classList`, `innerHTML`/`innerText`, `addEventListener` with `function()` callback — all ES5-safe |
| Storage | `localStorage.getItem/setItem/removeItem` — ES5-safe |
| CSRF | Cookie `csrf_token` read via `indexOf` loop — no `.find()` or `.startsWith()` |
| Sanitization | Every `innerHTML` assignment wraps values in `sanitizeHtml()`/`escHtml()` |

### HTMX Interaction

- HTMX is used for server-rendered HTML partials (hx-get, hx-post, hx-target, hx-swap) in the template
- The primary scanning workflow is JS-driven via XHR, not HTMX — HTMX mainly handles UI partials and form interactions
- HTMX version: 2.0.4 (vendored) — will be tested on real warehouse devices and swapped to 1.9.x if incompatible

### Accessibility Notes

- Scanning input uses `inputmode="none"` to prevent on-screen keyboard on touch devices (warehouse scanners use hardware keyboard wedge)
- Login page matches existing pattern: `autocomplete="on"`, labeled inputs
- No ARIA attributes added — matches existing pattern
